# 统计分析 V3 方案

> 基于 Emby Pulse 结构分析 + 我们现有技术栈制定  
> 核心原则：每个页面只回答一个问题，口径统一，层级清晰

---

## 一、页面结构（3 页）

### 1. 总览页（`/stats`）

**回答的问题：** "我的 Emby 今天/这周整体怎么样了？"

```
┌─────────────────────────────────────────────┐
│  核心指标卡片（4 个，一排横排）                 │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐       │
│  │ 媒体  │ │ 播放  │ │ 活跃  │ │ 时长  │       │
│  │ 总量  │ │ 总次数 │ │ 用户  │ │ 总时长 │       │
│  └──────┘ └──────┘ └──────┘ └──────┘       │
├─────────────────────────────────────────────┤
│  播放趋势（一条线，只有播放时长，可切 30天/12周/12月）│
├─────────────────────────────────────────────┤
│  最近 7 天 Top 5 内容（按时长排，海报+标题+时长）    │
├─────────────────────────────────────────────┤
│  活跃用户 Top 5（按时长排，头像+名字+时长）         │
└─────────────────────────────────────────────┘
```

**口径：**
- 核心指标卡片：播放总次数 + 30 天活跃用户数 + 总播放时长
- 趋势图：只有"播放时长"一个维度，按天/周/月聚合
- Top 内容：按时长排，不分电影剧集
- 用户榜：按时长排

**为什么这样：**
- 第一眼看 4 个数，3 秒了解全局
- 趋势图只有一条线，不搞双轴，干净
- 下面两栏是"快速发现"，让用户知道最近热门

### 2. 内容分析页（`/stats/content`）

**回答的问题：** "我的库里哪些内容最火？哪些没人看？"

```
┌─────────────────────────────────────────────┐
│  筛选区                                      │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐    │
│  │ 类型切换   │ │ 时间范围   │ │ 排序方式   │    │
│  │ 全部/电影  │ │ 30天/90天 │ │ 时长/次数  │    │
│  │ /剧集     │ │ /全部     │ │           │    │
│  └──────────┘ └──────────┘ └──────────┘    │
├─────────────────────────────────────────────┤
│  内容排行榜（表格）                            │
│  ┌──────────────────────────────────────┐   │
│  │ # │ 海报 │ 标题 │ 播放次数 │ 总时长 │ ... │   │
│  │ 1 │  🖼  │ XXX │   45    │ 32h   │     │   │
│  │ 2 │  🖼  │ YYY │   38    │ 28h   │     │   │
│  │...│      │     │         │       │     │   │
│  └──────────────────────────────────────┘   │
├─────────────────────────────────────────────┤
│  点击某行 → 底部展开该内容详情抽屉              │
│  ├── 趋势（该内容每周播放趋势）                  │
│  ├── 观看用户分布（谁看了、看多少）              │
│  └── 单集列表（如果是剧集，显示分季/分集）       │
└─────────────────────────────────────────────┘
```

**口径：**
- 主排序：默认按时长降序
- 可切换：按播放次数降序
- 内容聚合：剧集默认按作品级（Series）聚合，不是按单集
- 详情抽屉：点击后才展开，不跟排行抢主位

**为什么这样：**
- 顶部筛选区让用户自己控制"看什么"
- 排行榜是核心内容，一屏一个重点
- 详情抽屉是"下钻"逻辑，不影响主列表

### 3. 用户分析页（`/stats/users`）

**回答的问题：** "谁在看？看什么？怎么看的？"

```
┌─────────────────────────────────────────────┐
│  用户排行（按时长排，头像 + 名字 + 时长 + 次数） │
│  ┌──────────────────────────────────────┐   │
│  │ #1  👤  张三    45h  │  120 次播放    │   │
│  │ #2  👤  李四    38h  │   95 次播放    │   │
│  │ #3  👤  王五    22h  │   68 次播放    │   │
│  │ ...                 │               │   │
│  └──────────────────────────────────────┘   │
├─────────────────────────────────────────────┤
│  点击某用户 → 底部展开用户画像抽屉              │
│  ├── 3 个 KPI：总播放 / 沉浸时长 / 单次平均     │
│  ├── 成就徽章（深夜修仙、带薪观影、全平台制霸等）│
│  ├── 内容偏好（电影党/追剧党 + Top 最爱）       │
│  ├── 时段热力图（每小时活跃分布）               │
│  ├── 设备分布（客户端 + 设备类型）              │
│  └── 最近播放记录（海报+标题+时间）             │
└─────────────────────────────────────────────┘
```

**口径：**
- 用户排序：统一按时长降序
- 次要信息：显示播放次数作为补充
- 详情抽屉：不提前加载，点才加载

**为什么这样：**
- 用户排行是唯一主内容，不混其他维度
- 用户画像抽屉是 Emby Pulse 的亮点（成就徽章），值得借鉴
- 不在主列表里塞太多用户详情，保持列表干净

## 二、数据来源

### 方案 A：API 模式（推荐）

使用 Emby Playback Reporting 插件的 API：
- `GET /emby/user_usage_stats/` — 播放统计
- `GET /emby/Items/Counts` — 媒体库总量
- `GET /Users` — 用户列表
- `GET /Users/{id}/Items` — 内容详情

**优点：** 部署简单，不依赖挂载数据库
**缺点：** API 查询性能不如直读 SQLite

### 方案 B：SQLite 直读模式

挂载 Emby 的 `playback_reporting.db`，直接 SQL 查询。

**优点：** 性能最高，支持复杂聚合
**缺点：** 需要挂载数据库文件

### 建议

优先做 **API 模式**，因为我们现有架构已经是 API 模式。后续再加 SQLite 直读作为性能优化选项。

## 三、后端 API 设计

### 总览页

```
GET /api/v1/stats/overview
  → 返回：{ library_counts, total_plays, active_users_30d, total_duration_hours }

GET /api/v1/stats/trend?period=30d|12w|12m
  → 返回：{ "2026-03-01": 12345, "2026-03-02": 23456, ... }  （播放时长，秒）

GET /api/v1/stats/top-content?limit=5&period=7d
  → 返回：[{ item_id, name, type, poster_url, play_count, total_duration }]

GET /api/v1/stats/top-users?limit=5&period=7d
  → 返回：[{ user_id, user_name, avatar_url, total_duration, play_count }]
```

### 内容分析页

```
GET /api/v1/stats/content-rankings?type=all|movie|series&period=30d|90d|all&sort=duration|count&page=1&size=20
  → 返回：{ total, items: [{ item_id, name, type, poster_url, play_count, total_duration }] }

GET /api/v1/stats/content/{item_id}
  → 返回：{ trend: {...}, viewers: [...], episodes: [...] }  （点击展开详情）
```

### 用户分析页

```
GET /api/v1/stats/user-rankings?period=30d|90d|all&page=1&size=20
  → 返回：{ total, items: [{ user_id, user_name, avatar_url, total_duration, play_count }] }

GET /api/v1/stats/users/{user_id}
  → 返回：{ kpis, badges, preference, hourly_heatmap, devices, recent_plays }
```

## 四、前端路由

```
/stats         → StatsOverviewPage（总览）
/stats/content → StatsContentPage（内容分析）
/stats/users   → StatsUsersPage（用户分析）
```

不再用 `StatsPage` 做容器 + tab 切换，而是：
- 三个独立页面，各自独立加载
- 侧边栏直接指向对应路由
- 移动端底部 tab 也直接指向对应路由

---

## 五、实施顺序

1. **后端 API** — 先写 overview / trend / top-content / top-users
2. **前端总览页** — 4 个指标卡片 + 趋势图 + Top 5 × 2
3. **后端补充** — content-rankings / user-rankings / content-detail / user-detail
4. **前端内容分析页** — 筛选区 + 排行表 + 详情抽屉
5. **前端用户分析页** — 用户排行 + 用户画像抽屉（含徽章）
6. **提交推送**

## 六、不做的事（这次不做）

- ❌ 画质盘点（insight）— 这是独立功能，不混在统计里
- ❌ 报表生成（日报/周报/月报）— 后续独立做
- ❌ 热力图（时段分析放用户画像抽屉里，不单独做页面）
- ❌ 飙升榜/完播榜/长尾榜 — 数据支持不够，后续再做
- ❌ SQLite 直读模式 — 先把 API 模式做干净
